EventStorming!

  • Big Picture ES - to discover domains etc.
  • Process-Level ES - to discover Bounded Contexts
  • Design-Level ES - for a tactical DDD within a BC
  • Golden rule: events (verbs, past tense), NOT data structures (nouns)
  • Commands (change the system's state) & queries (views)
Intuitive: How was your evening? We don't answer like "wine, voodka, beer", but rather "I drunk too much". Events. And commands are like (depending on the adapter): "come on, just one more drink", "don't you wanna drink with me"

Tactical DDD





Specification, Policy
  • entity = unique id
  • aggregate = aggregates (encapsulates) invariants, rules to be fulfilled alltogether
  • repo = stores and restores an aggregate - single db transaction
  • service = stateles domain logic, e.g. calculators
  • vo = data structure identifiable based on props; money
  • event = not necesarry from event sourcing, but a noteworthy thing happening in the domain and announced either internally or also externally
  • snapshot = not pure DDD, a little trick which helped me a lot implementing; X-ray photo

Clean Architecture

In the center we have our business invariants, our rules we want to build on
This arch helps postponing the decision about the actual implementation Esencja = odłożenie decyzji o implementacji jak najdalej w czasie. Komponent - bounded context. Tam wilk/świnka jest jednoznaczne i zrozumiałe w ten sam sposób przez wszystkich użytkowników. To nie jest wilk z Czerwonego Kapturka np.
Ten Excel też jest port i adapter jak na rysunku (kulka i owijka wokół niej)
Shared - mentioned interfaces.
Domain - unit tests, in-out, descriptive (DSL); even business can understand.
App - use cases/scenarios, User Story, Kotlin's nice forthat. Tests - commands & operating on the business physics, aggregates, services. I use mocks and checks if command results in repo save.
Adapters - adapters for ports from domain and app + input and output ones; package-private access between modules. Here we add frameworks (e.g. Java Spring), integration testing (Spring test app). App properties, etc.
Monolith - main class, but instead of it we could have a couple of microservice apps and each should take e.g. adapters from another BC.
